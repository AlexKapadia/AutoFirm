## 2. Faithful structured summary

### 2.1 On-demand prices — verbatim (C1)

> **UNIT NOTE (critical):** the Bedrock pricing page (C1) on the 2026-06-17 render presents these per **1,000,000 tokens (per 1M)**. Historically Bedrock published a **per-1,000-token (per 1K)** model; AWS docs and many SDK/console surfaces still reference per-1K. **W5 MUST store the explicit unit and normalise — never assume per-1M.** (See RED flag §4.)

**Anthropic Claude on Bedrock** (per 1M tokens, on-demand):
```
Claude 3.5 Sonnet    (Public Extended Access, effective 1 Dec 2025)
    Input        : $6.00      Output       : $30.00
    Batch Input  : $3.00      Batch Output : $15.00

Claude 3.5 Sonnet v2 (Public Extended Access, effective 1 Dec 2025)
    Input        : $6.00      Output       : $30.00
    Batch Input  : $3.00      Batch Output : $15.00
    Cache Write  : $7.50      Cache Read   : $0.60
```
> These Bedrock rates **differ from the native Anthropic API prices** for the same model — Bedrock is a distinct price surface. Always price Bedrock invocations from the **Bedrock** price map, never the native-provider map.

**Amazon Nova models:** `unverified` — C1 showed the section headers ("Understanding Models", "Creative Content Generation models", "Speech Understanding and Generation Models") but the numeric rates **did not render** in the fetch. Snapshot directly from C1 before use; do not invent.

### 2.2 Batch inference discount — verbatim (C1)
> "Amazon Bedrock offers select foundation models (FMs) from leading AI providers ... for batch inference at a **50% lower price** compared to on-demand inference pricing."

Confirmed: **Batch = 50% of on-demand** (consistent with the Claude batch rows: $3/$15 vs $6/$30).

### 2.3 Provisioned Throughput — verbatim (C1)
> "For Provisioned Throughput pricing, please reach out to your account team" (for Anthropic models).

Provisioned Throughput is billed by **model units reserved per hour** (a time-based, commitment axis: no-commitment / 1-month / 6-month), **not per token**. This is a **different billing axis entirely** (see RED flag §4). Exact $/model-unit/hour is **unverified** from C1 (account-team gated for Anthropic); capture per model when provisioning.

### 2.4 `TokenUsage` (Converse API) field names — verbatim (C2)
Returned in Converse / ConverseStream responses. **Token counts only — no cost field.**
```
inputTokens          : "The number of tokens sent in the request to the model."          [Required: Yes]
outputTokens         : "The number of tokens that the model generated for the request."   [Required: Yes]
totalTokens          : "The total of input tokens and tokens generated by the model."      [Required: Yes]
cacheReadInputTokens : "The number of input tokens read from the cache for the request."   [Required: No]
cacheWriteInputTokens: "The number of input tokens written to the cache for the request."  [Required: No]
cacheDetails         : "Detailed breakdown of cache writes by TTL. Empty if no cache
                        creation occurred. Sorted by TTL duration (1h before 5m)."         [Array; Required: No]
```

### 2.5 InvokeModel invocation metrics (C3, C4 + AWS docs)
- **InvokeModel response headers** (Anthropic Claude): `x-amzn-bedrock-input-token-count` and `x-amzn-bedrock-output-token-count`. Bodies may carry an `amazon-bedrock-invocationMetrics` block with `inputTokenCount`, `outputTokenCount`, and cache fields (`cacheReadInputTokenCount`, `cacheWriteInputTokenCount`) plus latency.
- **Model-invocation logging** (C3) — opt-in, off by default; logs to **CloudWatch Logs and/or S3**. Log entry (`schemaType: "ModelInvocationLog"`) carries verbatim:
  ```
  input.inputTokenCount   : "The number of input tokens in the request."
  output.outputTokenCount : "The number of output tokens in the response."
  requestId, operation (Converse|InvokeModel), modelId, identity.arn, timestamp,
  requestMetadata (the ONLY caller-supplied field — key/value tags for cost attribution)
  ```
  **No dollar/cost field in the log entry.** Attribution is by `identity.arn` or `requestMetadata`.
- **CloudWatch runtime metrics** (C4): `InputTokenCount`, `OutputTokenCount`, `CacheReadInputTokens`, `CacheWriteInputTokens`, `InvocationLatency`. Again **token counts, never cost.**

### 2.6 Billed-vs-quota quirk — verbatim (C4)
> **"You're only billed for your actual token usage."** Quota *burndown* ≠ billing. Claude 3.7+ has a **5× output burndown** for *quota*, but billing uses raw counts:
> "if you use Anthropic Claude Sonnet 4 and send a request containing 1,000 input tokens and it generates a response equivalent to 100 tokens: **1,500 tokens** (1,000 + 100×5) will be depleted from your TPM and TPD quotas. **You'll only be billed for 1,100 tokens.**"
- Quota: `InputTokenCount + CacheWriteInputTokens + (OutputTokenCount × burndown_rate)`.
- **Billing base: raw `inputTokens` / `outputTokens` (+ cache read/write priced separately).** `CacheReadInputTokens` do **not** count toward quota.

### 2.7 Cost reconciliation
- **The API returns no cost.** Authoritative dollar cost lands in the **AWS Cost & Usage Report (CUR)** and **Cost Explorer**. Per-request metadata tagging (`requestMetadata`) and `identity.arn` enable cost-allocation breakdowns; combine with model-invocation logs for per-call attribution.

## 4. Best parts to take → W5 design

1. **Exact cost = f(provider-returned usage, versioned frozen price snapshot).**
   `cost = Decimal(price) / Decimal(unit_divisor) * Decimal(token_count)` where `unit_divisor ∈ {1_000, 1_000_000}` is **stored explicitly per price row.** `decimal.Decimal` throughout.

2. **`cost_source` provenance:** Bedrock invocations are `price_map_computed` from the frozen Bedrock map. A separate `provider_reported` reconciliation channel exists via **CUR/Cost Explorer** — W5 should support importing CUR to *reconcile* computed cost against AWS-billed cost (provenance = `provider_reported` for the CUR figure).

3. **Decompose `TokenUsage` correctly:**
   ```
   fresh_input  = inputTokens                       # Bedrock inputTokens already EXCLUDE cache-read
   output       = outputTokens
   cache_read   = cacheReadInputTokens   (priced at Cache Read rate, e.g. $0.60/1M)
   cache_write  = cacheWriteInputTokens  (priced at Cache Write rate, e.g. $7.50/1M)
   cost = fresh_input*in_rate + output*out_rate + cache_read*read_rate + cache_write*write_rate
   ```
   (Confirm whether `inputTokens` is inclusive or exclusive of cache-read per model family before shipping; C4 treats `InputTokenCount` as *excluding cached tokens*. **Differs from Gemini, where `promptTokenCount` INCLUDES cached** — do not share decomposition logic across providers.)

4. **RED — per-1K vs per-1M unit difference (Bedrock):** AWS has published Bedrock prices **per-1,000 tokens** historically; the current page renders **per-1M**. A naive flat divisor (assuming the Gemini per-1M convention) would be **off by 1000×**. W5 MUST store `unit_divisor` per price row and normalise; never assume the unit.

5. **RED — Provisioned Throughput is TIME-based, not token-based.** PT bills **$/model-unit/hour** regardless of tokens processed. W5 cannot price PT invocations by multiplying tokens × rate — it must model PT as a separate **time-rate cost stream** (units × hours × hourly rate), reconciled against CUR. Tag invocations with `billing_mode ∈ {on_demand, batch, provisioned_throughput}`.

6. **RED — billing ≠ quota burndown.** Do NOT use the 5× output burndown (a *quota* construct, Claude 3.7+) for *cost*. Cost uses raw `inputTokens`/`outputTokens`. Conflating the two over-states output cost 5×.

7. **Bedrock rates ≠ native provider rates.** Same model (e.g. Claude) is priced differently on Bedrock vs the native Anthropic API. Key the price map by **`(surface=bedrock, model_id, region?, tier, modality)`**; never reuse a native-provider rate for a Bedrock call.

8. **Reconciliation channel:** persist `requestId`, `modelId`, `identity.arn`, and optional `requestMetadata` tags so computed cost can be reconciled line-by-line against CUR / Cost Explorer; model-invocation logging (CloudWatch/S3) is the per-call audit source but is **off by default** — flag that W5 deployments must enable it for full auditability.

9. **Snapshot discipline:** freeze a dated, unit-stamped Bedrock price-map version; price historical invocations against the snapshot in force at invocation time.
