# BEST-PARTS — SCOR

## What AutoFirm ADOPTS

1. **SCOR is the reference operating model AutoFirm instantiates for any physical-goods company.**
   For panel rows that move atoms (Discrete manufacturing, E-commerce/DTC, Restaurant/food
   service), AutoFirm's business-build engine should scaffold the operation as **Plan -> Source ->
   Make -> Deliver -> Return**, with **Enable** as the cross-cutting governance layer. This is a
   ready-made, standard process decomposition — exactly what L2.B2 (function-decomposition) needs.
   - **Build implication:** L2.B11's operations playbook ships a SCOR template; a physical-industry
     client company is decomposed into the six SCOR processes by default, then parameterized.

2. **The five performance attributes are AutoFirm's operations KPI contract.** Every physical-ops
   client gets a dashboard with one+ metric per attribute: Reliability (Perfect Order Fulfillment),
   Responsiveness (Order Fulfillment Cycle Time), Agility, Cost (Cost to Serve), Asset Efficiency
   (Cash-to-Cash Cycle Time). These are real, computable-from-public/operational data KPIs that feed
   the modeling toolkit (L2.B4) and evidence/ charts.

3. **"Enable" = governance baked into the operating model.** SCOR puts risk/compliance/data/business
   rules in a dedicated process. This matches AutoFirm's fail-closed, audited-by-default doctrine
   (CLAUDE.md §5.6) — operations governance is not an afterthought, it is a first-class process.

## What AutoFirm REJECTS / caution
- **Reject SCOR for pure-digital businesses without adaptation.** SaaS/digital marketplaces have no
  "Make/Source/Return" of physical goods; forcing SCOR there is overfitting. Use SCOR only for the
  physical/hybrid panel rows; use SaaS (source 09) / marketplace (source 10) models for digital.
- Caution: SCOR has >150 KPIs — AutoFirm must select a *small, decision-relevant* subset per client
  (one metric per attribute), not dump all 150 (simplicity, CLAUDE.md §5.2).

## Quantification for evidence/
- A per-attribute KPI scorecard (5 numbers) per physical-ops client, computed from real public/
  operational data, with benchmark bands — directly chartable.
- Demonstrates generalization: the SAME six-process template + five attributes produces a sensible
  operating model for manufacturing, DTC retail, AND food service (3 panel rows) — a B12 generality
  proof point.
