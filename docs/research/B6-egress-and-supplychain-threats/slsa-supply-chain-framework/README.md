# SLSA — Supply-chain Levels for Software Artifacts

## 1. Full Citation

- **Title:** SLSA — Supply-chain Levels for Software Artifacts (framework specification)
- **Author / Org:** Open Source Security Foundation (OpenSSF), under the Linux Foundation (originated at Google; vendor-neutral)
- **Year:** v1.0 framework (current as of 2025)
- **Primary URL:** https://slsa.dev/
- **Corroborating references used:** Wiz Academy "What is the SLSA Framework?" (https://www.wiz.io/academy/application-security/slsa-framework); arXiv 2409.05014 "Analyzing Challenges in Deployment of the SLSA Framework for Software Supply Chain Security."

## 2. Faithful Structured Summary

SLSA ("salsa") is a **vendor-neutral framework for verifying how software artifacts are built and whether they have been tampered with**. It defines standards a build system must meet to produce **verifiable, trustworthy outputs**, organized into **tracks** and **levels**. The **Build track** has four levels (L0–L3) of increasing assurance:

| Level | Requirement (faithful) | Threat it counters |
| --- | --- | --- |
| **Build L0** | No SLSA present; no requirements met. | — (baseline, no guarantees) |
| **Build L1** | **Documented provenance** of the package's build process. | Mistakes; basic traceability of how an artifact was produced. |
| **Build L2** | **Cryptographically signed provenance, generated by a hosted build system.** Associates identities/systems with the software. | **Detects tampering *after* the build.** |
| **Build L3** | **Hardened, isolated build platform**; security enforced at the individual build; provenance is authenticated and unforgeable. | **Detects/prevents tampering *during* the build** and attacks against the build system itself. |

Central concept: **provenance** = signed, machine-verifiable metadata stating *what* was built, *from what sources*, *by which builder*. A consumer **verifies provenance before trusting/loading an artifact**; an artifact whose provenance is missing or fails verification is **not trusted**.

## 3. Best Parts to Take → AutoFirm controls

| SLSA finding | AutoFirm control it grounds |
| --- | --- |
| **Signed, verifiable provenance required before an artifact is trusted (L2/L3)** | Grounds AutoFirm's **signed-manifest verification *before* load** of any dynamically-loaded skill or MCP server. No provenance / failed verification → **refuse to load (fail-closed)**, never quarantine-and-run. |
| **L3 build isolation = artifact you load was produced in a hardened, tamper-evident pipeline** | Sets the bar for *trusted* skill/MCP sources: prefer artifacts built to L2+/L3 provenance over unverified registry pulls (counters MCP "rug pull" — see MCP folder). |
| **Provenance pins *what* and *from what sources*** | Grounds **version-pinned manifests** + allowlist: AutoFirm pins exact artifact + digest, and provenance is checked against the pin so a silently-swapped version fails verification. |
| **Levels are a graduated assurance ladder** | Lets AutoFirm set a **minimum SLSA level** per capability class (e.g. consequential-egress skills require higher assurance than read-only ones) — least-privilege applied to the supply chain. |
